# 设计理念：极简主义

*   **专注**：做好一件事。
*   **清晰**：代码应该能自我解释。
*   **克制**：少即是多 (Less is more)。

## 编码准则

1.  **可读性优于取巧**：首先为人类读懂而编写代码。
2.  **最少依赖**：尽可能依赖语言自带的标准库。
3.  **优雅的错误处理**：显式且优雅地处理运行失败。

> "简单是终极的复杂。" (Simplicity is the ultimate sophistication.)

## UI 设计准则

1.  **最小化点击次数**：相比嵌套结构（如多级下拉菜单），优先选择扁平化、一步直达的布局（如展开的侧边栏菜单或标签页列表），以降低用户的交互成本。

## AI 交互与输出规范

1.  **工件使用中文**：所有的 artifacts（如设计方案、总结文档、任务列表等）必须全部使用中文编写。
2.  **放慢思考步伐**：放慢思考和分析的步骤，不要一次性进行跨度太大的思考和跳跃式决策，保持逻辑连贯、稳步推进。
3.  **无痕执行（自动清理临时文件）**：对于在分析、调试或执行任务过程中临时创建的任何文件（如临时运行脚本、测试数据等），必须在任务执行结束或确认不再需要时，主动调用命令将其删除清理，始终保持项目工作区的绝对整洁。
4.  **环境销毁前确认**：在测试或调试容器时，若需要销毁容器（如执行 `docker compose down`）或清理内部网络，必须在操作前明确向用户发起咨询，在得到用户确认/批准后方可执行。
5.  **规范文档撰写**：Readme.md 文件撰写时必须内容具体、细节充实，同时语言保持精简，拒绝冗长啰嗦。

## 部署与 CI/CD 规范

1.  **热修复易失性警告**：服务器上的任何“热修复”（直接修改运行中容器内部文件）在执行 `start-prod.sh` 后都会丢失。这是因为生产环境脚本会直接从远端私有镜像库（如阿里云）拉取 `latest` 镜像并重建容器。
2.  **强制提交发布闭环**：任何被判定为有效的修复、优化或依赖追加（如 `requirements.txt` 更新、二进制文件 `wih` 加入），**必须**同步落库到本地源码。
3.  **标准化部署流程**：在生产服务器执行 `start-prod.sh` 更新前，必须确保本地代码已 `git commit` 并推送到远程仓库，等待 Github Actions 自动构建并推送最新镜像到云端注册表后，方可执行服务器更新。
4.  **构建配置前置审查**：每次推送 Github 触发 CI 自动构建阿里云镜像前，必须先严格检查并修正 `Dockerfile.prod` 文件，确保其中所有环境依赖、编译参数及镜像源等设置符合最新的代码和部署需求。
5.  **服务器登录信息**：公网vps/服务器的登陆信息存放在server.txt中。
6.  **Push前版本标签确认**：每次向远端推送（Push）代码前，必须主动询问用户是否需要打版本标签（Tag）以触发镜像构建流程。
7.  **代码修改后服务器测试确认**：每次修改代码后，必须主动询问用户是否需要在服务器端进行测试（该测试仅用于验证代码是否报错及满足要求，不涉及正式的 CI/CD 流程）。
