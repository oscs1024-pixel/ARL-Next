# 🤖 AI Agent 核心准则

> 处理本项目任务时，必须**严格遵守**以下规范。

## 1. AI 核心思维与行为红线
- **深度思考**: 前期的逻辑分析必须严谨详尽，**放慢分析步伐，保持逻辑连贯，禁止跳跃式决策**。
- **第一性原理**: 面对复杂需求或 Bug 时，禁止盲目依赖过往经验或生搬硬套现有模式（类比思维）。必须剥离表象，回归事物的最基础本质，将问题拆解为不可辩驳的基础事实，再从头进行逻辑推导与方案重构。
- **强制中文**: 生成的所有内容（包括对话回复与 Artifacts）必须使用中文。
- **无痕执行**: AI 在当前任务中**主动创建**的临时文件（如验证脚本、测试数据）在任务结束后**必须主动清理**，保持工作区整洁。
- **危险操作确认**: 涉及销毁容器、清理网络、放弃工作区修改或全局清除未追踪文件（如 `git reset --hard` / `git clean -fd`）等破坏性操作前，**必须向用户说明潜在影响并获批**。（注：清理上述“无痕执行”中 AI 自身创建的临时文件无需确认）。

## 2. 编码与架构设计规范
- **极简输出**: 保持克制，少即是多。
  - **代码极简**: 避免过度设计，尽量保持扁平化。
  - **UI 最小化**: 最小化点击次数。优先扁平化布局（侧边栏、标签页），**避免**深层嵌套以降低交互成本。
  - **精炼文档**: 更新文档需内容充实、语言精炼，禁止废话。
- **架构整洁**:
  - **职责单一**: 每个模块或函数只做一件事。
  - **清晰易懂**: 代码需自我解释。优先保证人类可读性，避免取巧写法。
  - **最少依赖**: 优先使用**标准库**，避免引入第三方包。
  - **错误处理**: 必须显式处理错误，禁止生吞异常。
- **性能优化**: 更新 Nginx 配置必须开启 Gzip，防止大体积静态资源明文传输拖慢加载。
- **代码审查方法论**:
  - **追溯完整调用链**: 禁止仅看局部逻辑。必须从最初入口完整追溯至最终出口（含沿途参数转换与下游消费逻辑）后再下结论。
  - **核实多调用方**: 函数被多处调用时，必须检查所有调用点上下文，不可偏盖全。

## 3. 部署与 CI/CD 闭环
- **Docker 镜像优化**: 修改或维护 `Dockerfile.prod` 等生产级构建文件时，必须强制使用**多阶段构建 (Multi-stage Build)**。在绝对保证系统环境与依赖稳定的前提下，将编译工具链与纯净运行时环境彻底隔离，最大化削减最终构建的镜像体积。
- **配置与凭证位置**: 
  - 版本信息存放在 `version.txt`。
  - 服务器 SSH 登录信息存放在 `server.txt`。
- **开发与提交闭环**: 
  - 任何有效修改必须同步至本地源码（注意：容器内的“热修复”在执行 `start-prod.sh` 后会重新拉取镜像而丢失）。
  - 修改完成后，需询问：“是否需打包热更新到服务器测试？”
  - 测试无误后，再询问是否更新 `version.txt` 版本并执行 `git commit`。
- **发布与部署前置检查**:
  1. 触发 Github CI 前：**必须检查并修正 `Dockerfile.prod`**。
  2. 代码 Push 前：**必须询问**：“是否需打 Tag 触发镜像构建？”
  3. 执行 `start-prod.sh` 前：**必须确保**代码已推送且最新镜像构建完成。
